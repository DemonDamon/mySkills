# 博客写作阶段详细规范

本文档是 Phase 2（Writing）的详细参考，补充 SKILL.md 中的写作流程。与 `/genblog` 指令共用同一套写作约束：物料自检后再写、强约束（语言/风格/结构/篇幅/插图/代码）。

## 1. 物料盘点模板

进入写作前，先输出物料盘点清单：

  - 02_xxx.md — <简述内容>
  ...

### 可用图片
- images/screenshot_xxx.png — <官方页面截图描述>（要求 ≥ 3 张）
- images/xxx.png — <爬取图片描述>

- 工程细节覆盖（含 10 问）：充足 / 需补充
- 代码片段可引用：充足 / 需补充
- 插图可用性（含官方截图）：充足 / 需补充
```

## 3. 写作风格指南

### 语言要求
- **中文为主**，术语首次出现时给出英文原文：`注意力机制（Attention Mechanism）`
- 避免翻译腔，用自然的中文表达
- 少用形容词和副词，多用动词和名词

### 信息密度
- 每段话必须有实质信息，删除"众所周知"等废话
- 一个概念一段话，不要混杂多个概念
- 关键数据/指标直接给出，不要说"效果有所提升"而应说"F1 提升了 3.2%"

### 过渡与衔接
- 章节间需有自然过渡（一句话即可）
- 用"为什么需要 → 是什么 → 怎么做 → 效果如何 → 生产落地注意事项"的逻辑链

### 代码规范
- **优先引用 clone 到本地的真实代码**（标注文件路径和行号）
- 只放核心片段（不超过 30 行/块）
- 必须有注释说明关键步骤
- 标明语言类型（python/bash/yaml 等）
- 伪代码优于完整代码（除非完整代码本身很短）

```python
# 示例：引用本地 clone 代码
# 文件: sources/github/<repo>/src/engine.py  L42-L58
def process_document(self, pdf_path: str):
    pages = self.extract_pages(pdf_path)     # 提取页面
    toc = self.detect_toc(pages[:20])        # 检测目录
    tree = self.build_tree(toc, pages)       # 构建树结构
    tree = self.enrich_nodes(tree)           # 丰富节点信息
    return tree
```

### 10 问分析的融入方式

10 问工程分析的结论不要单独成章再重复一遍，而应**自然融入**到以下章节：
- **工程实践**章节：融入部署、配置、性能调优相关问题的结论
- **生产落地评估**章节：融入适用边界、成本、安全、依赖风险的结论
- **总结与展望**章节：融入局限性和改进方向的结论

## 4. 插图使用规范

### 图片引用格式

```markdown
![图片简述](images/filename.png)

*图 N：一句话说明这张图展示了什么*
```

### 图片选择优先级
1. **官方页面截图**（`images/screenshot_*.png`）— 最优先
2. 爬取的原始架构图/流程图/实验结果图（`images/` 目录）
3. **Gemini/闭环生成的信息图**（`images/diagram_*.jpg` 等）— 自绘图首选
4. **mermaid 图** — 仅当无法使用闭环生图时的 fallback，不滥用
5. 占位标记（`<!-- 待补充：xxx 图 -->`）

### 官方截图的使用场景

| 截图类型 | 使用位置 | 说明 |
|---------|---------|------|
| 官网首页 | 背景/引言 | 展示产品定位 |
| 架构图 | 整体架构章节 | 官方视角的系统设计 |
| Benchmark | 评测章节 | 官方性能数据 |
| Demo 效果 | 工程实践章节 | 真实使用效果 |

### Mermaid 图（Fallback）

当 Gemini/闭环生图不可用时，可用 mermaid 作为自绘图 fallback。使用场景与示例：

```markdown
​```mermaid
graph TD
    A[用户输入] --> B[Tokenizer]
    B --> C[Encoder]
    C --> D{Router}
    D -->|类型A| E[Expert-1]
    D -->|类型B| F[Expert-2]
    E --> G[Decoder]
    F --> G
    G --> H[输出]
​```
```

> 注意：mermaid 图应简洁明了，节点不超过 15 个。复杂系统拆分为多张图。

## 5. 三轮写作与自评 checklist（Round 3）

Round 1 产出 outline.md（大纲 + 图文配对），Round 2 产出 blog.md 初稿，Round 3 对初稿执行自评并修正。

### 自评 checklist（每项 1–10 分，低于 8 分则修正）

**内容质量**
- [ ] 标题准确反映核心内容（无标题党）
- [ ] 摘要/开头 3 句话能让读者判断是否值得读
- [ ] 每段有实质信息，无废话/套话（信息密度）
- [ ] 关键结论有数据/代码/引用支撑
- [ ] 术语一致（同一概念不换多种称呼）
- [ ] 10 问分析结论已自然融入正文（工程实践、生产落地、局限）

**结构质量**
- [ ] 层次清晰：h1/h2/h3 不超过 3 级
- [ ] 有总→分→总节奏；含「生产落地评估」章节
- [ ] 由粗到细、先总览后拆解；关键处有「为什么有效」

**视觉与代码**
- [ ] 核心章节（架构、评测、产品界面）有配图，非纯文字
- [ ] 代码块为真实引用（带文件路径），单块 ≤30 行
- [ ] 表格用于对比数据，非大段文字

**格式**
- [ ] Markdown 正确；图片相对路径；无残留占位符/TODO

执行方式：对 blog.md 逐项打分，任一低于 8 分则针对性修改该处，再落盘。
