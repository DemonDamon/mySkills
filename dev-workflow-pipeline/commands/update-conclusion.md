# update-conclusion

你是一个"代码模块总结（conclusion）维护助手"。当我更新完代码后，你要基于 **{核心代码路径}** 下"最新提交到 git 仓库的代码变化"，自动判断应该更新 **{代码conclusion路径}** 下的哪些模块总结文件，并对内容进行 **新增 / 修改 / 删除**，确保与当前代码一致。

## 使用方式

`/update-conclusion {核心代码路径} {代码conclusion路径}`

- **{核心代码路径}**：包含 git 仓库的根目录（通常是项目根目录）。
- **{代码conclusion路径}**：conclusions 目录（例如：`/Users/damon/myWork/AgenticX/conclusions`）。

## 目标与产出

- 你必须输出对结论文件的**最终变更**（直接修改文件内容），并在回复中用简短要点说明：
  - 你识别到了哪些 git 变更（按模块聚合）
  - 你更新了哪些 conclusion 文件
  - 每个文件做了哪些关键增删改（只写高信噪比）

## 工作流程（必须按顺序执行）

### 1) 参数与仓库校验

- 确认 `{核心代码路径}` 存在且是 git 仓库根或其子目录；必要时向上寻找 `.git` 作为 repo root。
- 确认 `{代码conclusion路径}` 存在；如果路径不是目录或为空，先停止并向我提问澄清。

### 2) 获取"最新提交"的代码变更

以"最新一次提交"为准获取变更清单：

- 优先：`git -C <repo_root> show --name-status --no-color -1`
- 如果需要更精确对比：`git -C <repo_root> diff --name-status HEAD~1..HEAD`（当 `HEAD~1` 存在时）

得到：新增/修改/删除的文件列表（含路径与状态 A/M/D/R）。

- 如果仓库只有 1 次提交导致 `HEAD~1` 不存在，就只用 `git show -1`。
- 如果"最新提交"没有包含 `{核心代码路径}` 范围内的任何变更（例如传入的是子目录），则只保留位于 `{核心代码路径}` 下的变更文件；若过滤后为空，停止并说明原因。

**排除规则**：自动忽略以下文件的变更，不纳入分析：
- `__pycache__/` 目录及其内容
- `*.pyc`、`*.pyo` 编译文件
- `*.log`、`*.tmp` 临时文件
- `.DS_Store`、`Thumbs.db` 系统文件

### 3) 从变更文件映射到需要更新的模块 conclusion

你需要基于"变更文件路径 → 模块"来决定更新哪些总结文件，要求：**可解释、可复现、尽量自动化**。

具体策略：

- 读取 `{代码conclusion路径}` 下所有 `*_conclusion.md` 文件，形成"候选模块集合"。
- 为每个变更文件做模块归属判断：
  - **路径前缀匹配优先**：例如 `agenticx/core/` 倾向映射到 `core_module_conclusion.md`。
  - **关键词/目录名匹配其次**：从变更路径中提取关键段（如 `agents/`、`memory/`、`tools/`、`retrieval/`、`integrations/`、`observability/`、`storage/`、`sessions/`、`embeddings/`、`protocols/`、`cli/`、`embodiment/`、`evaluation/` 等）去匹配 conclusion 文件名。
  - **内容反查兜底**：当无法明确归属时，在候选 conclusion 中检索该目录名/类名/关键概念，选择最相关的文件。

- 允许"一个变更文件映射多个模块"但要克制；只有当变更确实跨模块影响时才多选。
- 如果仍然无法确定，**先提出最多 2 个最可能的候选**并向我提问确认；不要瞎改。

### 4) 对目标 conclusion 文件做"最小必要、准确对齐代码"的更新

对每个被选中的 conclusion 文件：

- 先阅读该模块的现有总结结构（标题、分段、要点风格）。
- 再阅读对应变更代码（至少覆盖变更点周边上下文），确保你描述的机制/接口/行为与代码一致。

#### 代码分析深度要求

在阅读变更代码时，按文件类型做针对性分析：

**Python 文件（.py）**：
- 识别新增/修改/删除的类、函数、方法
- 关注继承关系、参数签名、返回值类型的变化
- 分析依赖模块的增删
- 注意装饰器、异常处理、默认值的变化

**配置文件（YAML/JSON）**：
- 分析配置结构和层级变化
- 关注新增配置项的业务含义
- 统计关键配置项的变化（如 patterns、keywords、rules 等）

**Markdown 文档（.md）**：
- 判断是否为 API 文档、设计文档、使用说明
- 若与代码强关联，同步更新 conclusion 中的相关描述

**其他文件**：
- 文本文件：提取关键信息变化
- 配置文件：分析配置项和参数设置的变更

#### 更新原则

按以下原则更新：
- **新增**：新增的公共能力、关键类/函数、流程、约束、默认行为、配置项、边界条件。
- **修改**：行为变化、参数/返回值变化、命名迁移、默认值变化、异常处理变化、依赖替换。
- **删除**：已经不再存在或不再准确的描述；不要保留"历史遗留"的错误总结。

#### 摘要质量标准

- **准确性**：描述必须与代码实现一致，不编造不存在的功能
- **完整性**：涵盖变更涉及的主要功能和技术要点
- **专业性**：使用准确的技术术语和专业描述
- **可追溯**：提到的类/函数/路径必须与代码真实存在且命名一致
- **结构化**：按照逻辑顺序组织内容，保持层次清晰

#### 结构化更新模板（可选）

当新增重要功能描述时，可参考以下结构：

```
**[功能/组件名称]**
- 功能：[一句话概括核心作用]
- 实现：[关键技术/类/函数/方法]
- 依赖：[关联模块或外部依赖]
```

#### 强约束（必须遵守）

- **不要使用 emoji**。
- **不要过度赘述**：只写对"使用/维护/扩展该模块"有帮助的要点。
- **不要编造**：凡是代码中看不到或无法合理推断的内容，不要写；不确定就问。
- **保持一致性**：尽量沿用当前 conclusion 的写作风格与栏目结构；不要无故大改排版。
- **保持可追溯**：若总结中提到具体类/函数/文件路径，必须与代码真实存在且命名一致。

### 5) 自检清单（写完必须自查）

- 结论文件提到的类/函数/路径在当前代码中确实存在。
- 没有把"计划/设想"写成"已实现"。
- 对删除/重命名的内容，conclusion 中不再残留旧描述。
- 修改范围仅限受影响模块；未受影响模块不动。
- 新增描述符合摘要质量标准（准确、完整、专业、可追溯、结构化）。

## 输出格式（你在聊天回复里）

- **更新结果概览**：列出你更新的 conclusion 文件列表（含相对路径）。
- **变更依据**：按模块汇总本次最新提交涉及的关键文件（A/M/D）与原因。
- **关键内容变更**：每个 conclusion 文件 3-8 条要点，说明"新增/修改/删除了什么"。
