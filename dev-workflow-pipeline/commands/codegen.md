# codegen

把第三方优秀项目的**关键思想、机制与实现**，以"最小可迁移机制"的方式**可验证地内化**到 AgenticX，并对每个功能点提供**可运行的冒烟测试**落盘到 `tests/`。

---

## 使用方式

```
/codegen <upstream_name> [scope...]
```

| 参数 | 必选 | 说明 | 示例 |
|------|------|------|------|
| `upstream_name` | 是 | 上游项目名，对应 `research/codedeepresearch/<name>/` 目录 | `openclaw`, `agentscope`, `crewAI` |
| `scope...` | 否 | 限定本次要实现的功能范围。不填则按 proposal 中 P0→P1 顺序推进 | `memory tool router` |

用户也可以在对话中用自然语言覆盖任何默认值（如粒度、依赖策略等）。

---

## 约定路径（自动按 upstream_name 推导）

| 用途 | 路径模板 |
|------|---------|
| 研究目录 | `research/codedeepresearch/<upstream_name>/` |
| 元信息 | `research/codedeepresearch/<upstream_name>/meta.md` |
| 技术方案 | `research/codedeepresearch/<upstream_name>/<upstream_name>_proposal.md` |
| 上游源码 | `research/codedeepresearch/<upstream_name>/upstream/` 或同级目录下的源码文件夹 |

如找不到上述文件，在聊天中提示用户补充路径，不要猜测。

---

## 默认约束（直接使用，不必每次确认）

以下参数均有默认值，**直接使用默认值开始工作**。仅当用户在对话中明确要求覆盖时才调整。

| 参数 | 默认值 |
|------|-------|
| 推进顺序 | 按 proposal 中 P0→P1→P2 顺序；有 `scope` 参数时只做指定范围 |
| 交付粒度 | MVP（可运行、有测试、可合入） |
| 新增依赖 | 尽量不新增；若必须新增，先说明必要性与替代方案 |
| Python 版本 | 以仓库 `setup.py` / `pyproject.toml` 已有约束为准 |
| 质量优先级 | 可维护 > 可控 > 可验证 |

## 硬性验收（必须满足）

- **每一个功能点**都必须有对应的**冒烟测试**落盘到 `tests/`，并能在本仓库中通过执行。
- **优先复用成熟上游实现**：若上游已有稳定模块/实现，允许复制/改造集成，但必须：
  - 保留关键语义一致性（行为/边界条件）
  - 处理好许可证与引用（在文件头或 `NOTICE`/注释中标注来源、路径、必要时保留版权声明）
- **代码可读可维护**：新增模块需有清晰命名、类型/注释、错误处理、最小公共接口与文档字符串。
- **变更可控**：避免大范围重构；以"小步可验证"方式逐个功能点合入。

## 执行流程（严格按顺序）

你是一个自主编码智能体。按以下步骤在 AgenticX 仓库内完成实现与测试落盘。

### 1) 读取上游研究成果

按约定路径读取：
1. `meta.md` — 了解上游项目基本信息（语言、许可证、仓库结构）
2. `<upstream_name>_proposal.md` — 获取已分析好的技术方案与功能点分级
3. 上游源码（按 proposal 中引用的关键文件按需读取，不要一次性全部读入）

### 2) 生成可执行功能点清单

在聊天中列出功能点表格（不要只在脑子里想）：

| # | 功能点 | 优先级 | 上游证据（文件/类/函数） | AgenticX 落点（文件/类/函数） | 验收场景 |
|---|--------|--------|------------------------|-------------------------------|---------|

- 如果用户传入了 `scope...`，只列出 scope 覆盖的功能点
- 如果 proposal 中已有清晰的功能点分级，直接复用，不必重新分析

### 3) 设计最小接口（先定边界再写代码）

对当前批次要实现的功能点，先给出最小接口草案（以 AgenticX 现有风格为准）：
- 输入/输出数据结构
- 错误模型（抛什么异常/返回什么错误对象）
- 可观测性（必要日志/trace hook 的放置点）
- 扩展点（registry / hook / strategy interface）

接口不确定时，优先"薄封装 + 适配层"，避免侵入式改动。

### 4) 实现

对每个功能点选择策略（并说明理由）：
- **直接复用**：复制上游实现并做最小适配（保留行为一致性）
- **抽象迁移**：迁移关键机制/接口，重新实现细节以更贴合 AgenticX
- **轻量替代**：上游过重/耦合太强时，做最小等价机制，明确差异与风险

实现要求：
- 每个功能点单独一个小变更集：代码 + 对应测试同时落盘
- 关键路径加日志（不要刷屏，信息密度高）
- 失败要可诊断（错误信息包含上下文）

### 5) 冒烟测试（每个功能点必须有，且可运行）

- 落盘位置：`tests/`
- 命名规范：`tests/test_smoke_<upstream_name>_<feature>.py`
- 每个功能点至少 1 个冒烟用例，覆盖：
  - 正常路径（Happy path）
  - 至少 1 个失败/边界路径

测试编写原则：
- 不依赖外部网络；如必须依赖，提供 mock/stub 或显式跳过
- 运行要快（默认单测级别）
- 断言要稳定（断言结构/关键字段，不断言易变字符串）

执行方式（你需要真的运行并修到通过）：
- `pytest -q tests/test_smoke_<upstream_name>_<feature>.py`
- 或批量：`pytest -q -k "smoke_<upstream_name>"`

### 6) 逐个推进，先绿再扩

- 先完成当前功能点（含测试通过）→ 再推进下一个
- 任何功能点测试不过：停止扩张范围，先修到绿
- 顺序：P0 全部完成 → P1 → P2

## 汇报要求（聊天中必须包含）

每完成一个功能点，用清单汇报：

- **实现了什么**：功能点名称 + 行为描述
- **改了哪些文件**：列出路径（新增/修改）
- **上游对齐情况**：对齐 / 部分对齐 / 不对齐（并解释差异）
- **如何运行测试**：给出具体 `pytest` 命令

全部结束后给出总结：

- 已完成的功能点列表（含优先级）
- 未完成与原因
- 建议的下一步（明确到 AgenticX 模块/目录级别）
