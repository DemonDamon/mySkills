# codeview

对指定目标做一次完整的 Code Review，根据变更类型（feature / bugfix / refactor / perf / infra / hotfix）自动调整审查权重，产出按严重级别排序的审查报告，并可选地自动修复、本地验证、提交与飞书文档归档。

支持三种审查模式：**分支模式**、**工作区模式**、**文件模式**。

---

## 使用方式

```
/codeview <target> [options...]
```

`<target>` 可以是以下任一形式：

| 形式 | 模式 | 说明 |
|------|------|------|
| `<branch>` | 分支模式 | 审查分支相对基准的增量改动 |
| `--working` | 工作区模式 | 审查当前工作区未暂存的改动（`git diff`） |
| `--staged` | 工作区模式 | 审查已暂存的改动（`git diff --cached`） |
| `path/to/file.py` | 文件模式 | 审查指定文件的当前内容 |
| `path/to/folder/` | 文件模式 | 审查文件夹下所有代码文件 |
| `file1.py file2.py dir/` | 文件模式 | 批量审查多个文件/文件夹 |

**模式判定规则**（按优先级）：
1. 包含 `--working` 或 `--staged` → 工作区模式
2. 目标中有任一路径在文件系统中存在（文件或目录） → 文件模式
3. 其他 → 分支模式

示例：
- `/codeview v0.1` — 审查 v0.1 相对 main 的改动（分支模式）
- `/codeview feature/auth --base=develop` — 以 develop 为基准（分支模式）
- `/codeview --working` — 审查所有未暂存改动（工作区模式）
- `/codeview --staged` — 审查已暂存改动（工作区模式）
- `/codeview --working --type=feature` — 按新功能模式审查工作区改动
- `/codeview agenticx/core/overflow_recovery.py` — 审查单个文件（文件模式）
- `/codeview agenticx/llms/` — 审查整个文件夹（文件模式）
- `/codeview agenticx/core/prompt.py agenticx/core/handoff.py tests/test_smoke_openclaw_prompt_mode.py` — 批量审查（文件模式）
- `/codeview --working -- agenticx/core/` — 仅审查该目录下的工作区改动（工作区模式 + 文件过滤）
- `/codeview v0.2 --fix --commit --env=terminal_intent` — 审查 + 自动修复 + 提交
- `/codeview hotfix/payment --type=hotfix --fix --commit` — 紧急修复模式
- `/codeview v0.1 --feishu` — 审查后将报告写入飞书文档
- `/codeview --working --spec=research/codedeepresearch/openclaw/openclaw_proposal.md` — 带需求规格审查工作区改动
- `/codeview agenticx/core/ --spec="实现上下文溢出的三级渐进恢复：L1截断工具结果>L2 LLM压缩>L3启发式丢弃"` — 带内联需求审查文件夹
- `/codeview v0.1 --spec=docs/PRD_auth.md --type=feature` — 带 PRD 文档的功能审查

## 参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `<target>` | 分支名 / `--working` / `--staged` / 文件路径 / 文件夹路径（可多个） | 无，必须提供 |
| `--base=<branch>` | 基准分支（仅分支模式） | `main` |
| `-- <paths...>` | 文件过滤：限制审查范围到指定路径（仅分支/工作区模式） | 全部变更文件 |
| `--fix` | 审查后自动修复发现的问题（最小改动原则） | 不修复，仅报告 |
| `--commit` | 修复后自动提交（需配合 `--fix`） | 不提交 |
| `--push` | 提交后自动推送到远端（需配合 `--commit`） | 不推送 |
| `--env=<name>` | 用于本地验证的 conda 环境名 | 系统默认 Python |
| `--feishu` | 将审查报告写入飞书文档 | 不写入 |
| `--type=<type>` | 变更类型，影响审查权重分配。可选值见下方"变更类型"表 | 自动推断 |
| `--spec=<path\|text>` | 需求规格。文件路径（PRD / proposal / issue 等）或引号括起的内联文本。提供后启用需求对齐审查 | 不提供，仅做代码质量审查 |
| `--no-proxy` | 跳过代理设置（默认会检测并设置代理） | 自动检测 |

如果用户未提供任何 `<target>`，必须主动反问：
> 请提供审查目标，例如：
> - `/codeview v0.1` — 审查分支
> - `/codeview --working` — 审查工作区改动
> - `/codeview agenticx/core/` — 审查指定文件夹

## 变更类型与审查权重

### 变更类型定义

| 类型 | 含义 | 典型场景 |
|------|------|---------|
| `feature` | 新功能 | 新增接口、页面、模块、业务能力 |
| `bugfix` | 缺陷修复 | 修复线上/测试中发现的 bug |
| `refactor` | 重构 | 不改行为，改善代码结构 |
| `perf` | 性能优化 | 优化查询、算法、缓存策略 |
| `infra` | 基础设施 | CI/CD、Dockerfile、依赖管理、配置变更 |
| `hotfix` | 紧急修复 | 生产故障止血，最小改动原则 |

当用户未指定 `--type` 时，在 Phase 1 完成后根据 commit message、diff 内容和文件类型自动推断。推断逻辑：
- commit message 含 feat/feature/add → `feature`
- commit message 含 fix/bug/patch → `bugfix`
- commit message 含 refactor/rename/move → `refactor`
- commit message 含 perf/optimize/speed → `perf`
- 变更仅涉及 Dockerfile / CI / config / .yml → `infra`
- commit message 含 hotfix / urgent / emergency → `hotfix`
- 无法判断时默认 `feature`

### 审查权重矩阵

每个审查维度始终覆盖，但根据变更类型调整关注程度（高/中/低）：

| 维度 | feature | bugfix | refactor | perf | infra | hotfix |
|------|---------|--------|----------|------|-------|--------|
| 正确性 | 高 | 高 | 高(等价) | 中 | 中 | 高 |
| 安全性 | 高 | 中 | 低 | 低 | 高 | 中 |
| 健壮性 | 中 | 高 | 中 | 中 | 高 | 高 |
| 可维护性 | 高 | 低 | 高 | 中 | 中 | 低 |
| 性能 | 中 | 低 | 中 | 高 | 低 | 低 |
| 测试 | 高 | 高 | 中 | 高 | 低 | 中 |
| 规范性 | 中 | 低 | 中 | 低 | 中 | 低 |

权重为"高"时：该维度的发现应提升一级严重度（低→中、中→高）。
权重为"低"时：该维度的发现可降一级严重度，但安全类问题不降级。

### 按类型附加的审查关注点

- feature：是否有需求文档/PR 描述可对齐？API 设计是否向后兼容？是否影响已有功能？
- bugfix：根因分析是否充分？是治标还是治本？是否有回归测试？
- refactor：行为是否严格等价？重构方向是否使代码更简洁？已有测试是否仍覆盖？
- perf：是否有 benchmark/数据佐证？是否为了性能引入了过度复杂度？
- infra：环境变量是否安全？构建是否可重现？是否兼容所有目标环境？
- hotfix：改动是否最小？是否有回滚方案？事后是否需要补测试？

## 指令行为（严格按顺序执行）

### Phase 0：环境准备与模式判定

1. **模式判定**：按以下规则确定审查模式
   - 包含 `--working` → **工作区模式**（未暂存改动）
   - 包含 `--staged` → **工作区模式**（已暂存改动）
   - `<target>` 中任一路径在文件系统中存在 → **文件模式**
   - 其他 → **分支模式**

2. **需求规格加载**（仅当提供 `--spec` 时）：
   - 如果 `--spec` 值是一个存在的文件路径 → 读取文件内容作为需求规格
   - 如果 `--spec` 值不是文件路径（即内联文本） → 直接使用该文本作为需求规格
   - 解析需求规格，提取：
     - **功能性需求（FR）**：系统应该做什么（功能点、接口行为、业务规则、输入输出约定）
     - **非功能性需求（NFR）**：系统应该做到什么程度（性能指标、安全约束、可用性目标、并发要求、数据一致性、容错策略）
     - **验收条件（AC）**：如何判定需求已满足（测试场景、边界条件、预期行为）
   - 将提取结果整理为内部结构化清单（不输出给用户），供 Phase 2 审查使用
   - 如果需求文档过长（> 5000 字），提取关键章节摘要而非全文，避免上下文膨胀

3. **按模式执行环境准备**：

#### 分支模式
1. 代理检测：如果 `git fetch` 超时或失败，提示用户是否需要设置代理环境变量
2. 分支拉取：
   ```bash
   git fetch origin <branch>
   git checkout <branch>
   ```
3. 确认当前分支：`git branch --show-current`

#### 工作区模式
1. 确认当前分支：`git branch --show-current`
2. 检查工作区状态：
   ```bash
   git status --short
   ```
3. 如果无变更，提示用户：
   > 当前工作区没有未暂存/已暂存的改动，无需审查。

#### 文件模式
1. 验证所有目标路径存在（不存在的报错并列出）
2. 展开文件夹为文件列表：
   ```bash
   find <folder> -type f \( -name '*.py' -o -name '*.ts' -o -name '*.tsx' -o -name '*.js' -o -name '*.jsx' -o -name '*.go' -o -name '*.rs' -o -name '*.java' -o -name '*.rb' -o -name '*.sh' -o -name '*.yaml' -o -name '*.yml' -o -name '*.toml' -o -name '*.json' -o -name '*.sql' -o -name '*.md' \) | sort
   ```
3. 排除常见非代码路径：`node_modules/`, `__pycache__/`, `.git/`, `dist/`, `build/`, `.venv/`, `venv/`
4. 如果展开后文件数 > 50，提示用户确认：
   > 目标路径下有 N 个代码文件，全部审查预计较慢。是否继续？还是缩小范围？

### Phase 1：变更识别与类型判定

**按模式获取变更内容**：

#### 分支模式
1. 提交历史：列出目标分支相对基准分支的所有增量提交
   ```bash
   git log --pretty=format:'%h | %an | %ae | %ad | %s' --date=short <base>..<branch>
   ```
2. 作者识别：提取并汇总所有提交者（姓名、邮箱）
3. 变更文件清单：
   ```bash
   git diff --name-status <base>...<branch>
   ```
   如果指定了 `-- <paths...>` 过滤：
   ```bash
   git diff --name-status <base>...<branch> -- <paths...>
   ```
4. 变更统计：
   ```bash
   git show --stat --oneline <commits>
   ```

#### 工作区模式
1. 作者识别：当前 git 用户
   ```bash
   git config user.name && git config user.email
   ```
2. 变更文件清单：
   ```bash
   # --working 模式
   git diff --name-status
   # --staged 模式
   git diff --cached --name-status
   ```
   如果指定了 `-- <paths...>` 过滤：
   ```bash
   git diff --name-status -- <paths...>
   git diff --cached --name-status -- <paths...>
   ```
3. 变更统计：
   ```bash
   git diff --stat        # --working
   git diff --cached --stat  # --staged
   ```

#### 文件模式
1. 作者识别：从 git blame 提取各文件最近修改者
   ```bash
   git log -1 --format='%an <%ae>' -- <file>
   ```
2. 文件清单：Phase 0 展开的文件列表
3. 文件统计：
   ```bash
   wc -l <files>
   ```

**变更类型判定（所有模式通用）**：
5. 变更类型判定：
   - 如果用户通过 `--type` 指定，直接使用
   - 分支模式：根据 commit message、diff 内容、变更文件类型自动推断（规则见"变更类型与审查权重"节）
   - 工作区模式：根据 diff 内容和变更文件类型推断；无 commit message 可参考时默认 `feature`
   - 文件模式：根据文件内容和命名推断（`test_` 前缀 → 可能是 `feature` 或 `bugfix`；`Dockerfile` / `.yml` → `infra`）
   - 推断结果在报告中标注，并注明"[自动推断]"或"[用户指定]"

### Phase 2：逐文件审查

对每个变更/目标文件执行：

1. **获取审查内容**：
   - 分支模式：获取完整 diff
     ```bash
     git diff --unified=3 <base>...<branch> -- <file>
     ```
   - 工作区模式：获取工作区 diff
     ```bash
     git diff --unified=3 -- <file>          # --working
     git diff --cached --unified=3 -- <file>  # --staged
     ```
   - 文件模式：读取文件完整内容（非 diff，做全量审查）

2. **代码质量审查**（始终执行，按优先级，根据变更类型权重矩阵调整关注力度）：
   - 安全性：硬编码密钥/Token、SQL 注入、XSS、路径遍历（Path Traversal）、认证绕过、时序攻击、不安全的依赖版本
   - 正确性：逻辑错误、边界条件、异常处理缺失、资源泄漏、可变性误用（应不可变的地方用了可变对象）；refactor 类型额外检查行为等价性
   - 健壮性：异常兜底不足、输入校验、空值/空路径保护
   - 可维护性：重复代码、命名不清、过度耦合、缺少类型注解、公共 API 缺少文档字符串、单函数超 50 行、单文件超 800 行、嵌套深度超 4 层
   - 性能：不必要的全量扫描、N+1 查询、内存膨胀；perf 类型要求提供 benchmark 佐证
   - 测试覆盖：feature/bugfix 类型要求有对应测试；refactor 要求已有测试仍覆盖
   - 规范性：残留调试语句（print/console.log）、残留 TODO/FIXME、文件末尾换行、多余空行、import 顺序
   - 按类型附加关注点：参见"变更类型与审查权重"节的"按类型附加的审查关注点"

3. **需求对齐审查**（仅当提供 `--spec` 时执行）：
   对照 Phase 0 提取的需求清单，逐项检查：
   - **功能完整性**：需求中的每个功能点是否都有对应实现？缺失的功能点标记为高风险
   - **意图对齐**：实现方式是否与需求意图一致？（例：需求说"批量处理"，实现却是逐条循环；需求说"异步"，实现却是同步阻塞）
   - **非功能达标**：
     - 性能约束：需求中的延迟/吞吐/并发指标，代码层面是否有对应保障（索引、缓存、连接池、限流等）
     - 安全约束：需求中的权限/加密/审计要求是否落地
     - 可用性约束：需求中的容错/降级/重试策略是否实现
     - 数据一致性：需求中的事务/幂等/最终一致性保证是否到位
   - **边界覆盖**：需求隐含的边界场景是否处理（空输入、超大输入、并发冲突、网络中断等）
   - **验收条件覆盖**：需求中的验收条件是否有对应测试用例
   - **过度实现**：是否实现了需求未要求的功能（YAGNI 原则）

   需求对齐发现的严重级别判定：
   - 功能缺失 / 意图偏离 → 高风险
   - 非功能不达标 / 边界遗漏 → 中风险
   - 验收条件无测试覆盖 / 过度实现 → 低风险

4. 语法检查：对变更的 Python 文件执行 `python3 -m py_compile`

### Phase 3：产出审查报告

按以下格式输出（在聊天中），根据模式调整"基本信息"内容：

```
## Code Review 报告

### 基本信息
- 审查模式：分支模式 / 工作区模式(未暂存|已暂存) / 文件模式
- 目标：<branch> / 工作区改动 / <file1>, <file2>, ...
- 基准：<base>（分支模式）/ HEAD（工作区模式）/ N/A（文件模式）
- 变更类型：<type> [自动推断/用户指定]
- 提交人：<author(s)>
- 提交数：<N> 个（分支模式）/ N/A（工作区/文件模式）
- 审查文件：<N> 个（新增 A / 修改 M / 删除 D）（分支/工作区模式）
- 审查文件：<N> 个（共 <lines> 行）（文件模式）

### 需求对齐（仅当提供 --spec 时输出）

#### 需求规格来源
- 文件：<spec_path> / 内联文本

#### 功能性需求覆盖
| # | 功能点 | 状态 | 对应实现 | 备注 |
|---|--------|------|---------|------|
| FR-1 | <需求描述> | ✅ 已实现 / ⚠️ 部分实现 / ❌ 未实现 | <文件:行号> | <差异说明> |

#### 非功能性需求覆盖
| # | 约束 | 状态 | 代码保障 | 备注 |
|---|------|------|---------|------|
| NFR-1 | <约束描述> | ✅ 达标 / ⚠️ 存疑 / ❌ 未保障 | <文件:行号> | <差异说明> |

#### 验收条件覆盖
| # | 验收条件 | 有测试覆盖 | 测试位置 |
|---|---------|-----------|---------|
| AC-1 | <条件描述> | ✅ / ❌ | <test_file:行号> |

### 发现（按严重级别）

#### 🔴 高风险（必须修复）
- [文件:行号] 问题描述 + 影响 + 建议修复方式

#### 🟡 中风险（建议修复）
- [文件:行号] 问题描述 + 影响 + 建议修复方式

#### 🟢 低风险 / 改善建议
- [文件:行号] 问题描述 + 建议

### 正面评价
- 列出代码改进亮点（如安全增强、重复消除、架构优化等）

### 测试 / 验证缺口
- 缺少哪些自动化测试覆盖
```

### Phase 4：自动修复（仅当 `--fix` 时执行）

1. 修复原则：
   - 最小改动，不改业务逻辑
   - 不新增依赖（除非用户明确允许）
   - 保持原有代码风格与日志风格
   - hotfix 类型：只修复高风险问题，中/低风险留待后续迭代；需要在 commit message 中注明"事后需补测试"
   - refactor 类型：修复后需要确认行为等价，跑通已有测试
2. 修复流程：
   - 仅修复高风险和中风险问题（hotfix 仅高风险）
   - 每处修改说明：改了什么、为什么改
3. 语法验证：修复后重新编译检查
4. 本地行为验证：
   - 如果指定了 `--env`，使用对应 conda 环境：`conda run -n <env> python <script>`
   - 否则使用系统 Python
   - 运行相关入口脚本，确认无回归

### Phase 5：提交与推送（仅当 `--commit` 时执行）

适用条件：分支模式和工作区模式。**文件模式不支持 `--commit`**（因为没有 diff 概念，修复已直接写入文件）。

1. 提交门禁：如果仍有未修复的高风险问题，阻断提交并提示用户："仍有 N 个高风险问题未修复，建议先处理后再提交。确认强制提交请加 `--force`"
2. 暂存修复文件：`git add <files>`
3. 生成 commit message：
   - 格式：`fix(<scope>): <简要描述>`
   - body 列出每处修复点
4. 提交：`git commit`
5. 推送（仅当 `--push`）：
   - 分支模式：`git push origin <branch>`
   - 工作区模式：`git push origin HEAD`

### Phase 6：飞书文档归档（仅当 `--feishu` 时执行）

1. 调用飞书 MCP 创建文档，标题：`Code Review: <branch> (<date>)`
2. 写入审查报告内容（标题、正文、代码块）
3. 返回文档链接

## 信息不足时的反问策略

当用户提供的信息不足以执行时，主动反问而非猜测：

| 缺失信息 | 反问模板 |
|----------|---------|
| 未提供任何目标 | "请提供审查目标，例如 `/codeview v0.1`（分支）、`/codeview --working`（工作区）、`/codeview src/`（文件夹）" |
| 分支拉取失败（网络） | "git fetch 失败，是否需要配置代理？请提供代理地址（如 `http://127.0.0.1:7897`）" |
| 不确定基准分支 | "当前默认以 `main` 为基准，是否正确？如果是其他分支请用 `--base=<branch>` 指定" |
| 文件路径不存在 | "以下路径不存在：`<path>`。请检查路径拼写或提供正确路径" |
| 文件模式文件数过多 | "目标路径下有 N 个代码文件，全部审查预计较慢。是否继续？还是用更具体的路径缩小范围？" |
| 工作区无改动 | "当前工作区没有未暂存/已暂存的改动，无需审查。是否改用文件模式审查特定文件？" |
| 修复可能影响业务 | "发现 N 个问题，其中 M 个涉及业务逻辑，是否仅修复纯健壮性问题？还是全部修复？" |
| 需要验证但不知道环境 | "本地验证需要运行预测脚本，请问使用哪个 conda 环境？（如 `--env=terminal_intent`）" |
| 修复后是否提交 | "修复已完成并验证通过，是否需要提交到当前分支？" |
| 变更类型不明确 | "无法从 commit message 自动推断变更类型，请确认：这次改动属于 feature / bugfix / refactor / perf / infra / hotfix 中的哪种？" |
| spec 文件路径不存在 | "指定的需求规格文件不存在：`<path>`。请检查路径拼写，或用引号传入内联需求描述" |
| spec 内容过于模糊 | "需求规格中未能提取出明确的功能点或非功能约束，审查将仅做代码质量检查。如需需求对齐审查，请补充更具体的需求描述" |

## 注意事项

1. 只读优先：默认只产出报告，不做任何修改；修改需显式 `--fix`
2. 安全第一：不提交含密钥/凭据的文件；发现硬编码密钥时标记为高风险
3. 代理自动检测：首次 `git fetch` 超时时自动提示代理配置（仅分支模式）
4. Python 环境：conda 环境下优先用 `python`（非 `python3`），避免命中系统 Python
5. 不修改基准分支：所有操作都在目标分支上进行（分支模式）
6. 变更类型透明：报告中始终标注变更类型及其来源（自动推断/用户指定），方便回溯审查依据
7. 文件模式审查深度：文件模式做全量代码审查（非 diff），审查深度与 diff 模式一致，但额外关注文件整体结构
8. 路径过滤：分支模式和工作区模式支持 `-- <paths...>` 缩小审查范围到指定路径
9. 需求规格层级：`--spec` 是可选增强，不提供时审查仍然完整（覆盖代码质量七大维度）；提供后额外启用需求对齐审查，不替代原有维度
10. spec 内容保密：需求规格内容仅用于审查判断，不会出现在报告的"发现"描述中（避免敏感需求泄漏），仅在"需求对齐"章节以摘要形式引用
