# codegen-proceed

断点续做：当 codegen 执行被中断（网络、超时、IDE 重启、Agent 崩溃等），恢复到可验证的稳定状态，然后**继续推进未完成的功能点**。

---

## 步骤 1：自动恢复上下文（不要问用户，自己查）

按以下优先级查找"我上次做到哪了"：

| 优先级 | 方法 | 命令/操作 |
|--------|------|----------|
| 1 | Git 状态 | `git status`、`git diff --stat`、`git log --oneline -5` — 看最近改了什么文件、有没有未提交的变更 |
| 2 | 测试结果 | `pytest -q --tb=line` — 跑一遍现有冒烟测试，建立全局基线（哪些绿、哪些红） |
| 3 | 终端文件 | 读取 terminals 文件夹下最近的终端输出，找最后执行的命令和报错 |
| 4 | 文件时间 | 看 `tests/test_smoke_*.py` 和 `agenticx/` 下最近修改的文件 |

产出（在聊天中列出）：
- **已完成的功能点**：哪些冒烟测试已存在且通过
- **当前功能点状态**：正在做 / 代码写了但测试不过 / 还没开始
- **有红的测试吗**：列出失败的测试和错误摘要

---

## 步骤 2：修红（如果有失败）

**原则：最小修复，立刻回归。**

- 只改"能让当前失败消失"的地方，不顺手加功能/重构/美化
- 通常只需改 1~3 个文件（报错堆栈提到的文件）
- 改完立刻重跑失败的那条测试
- 单条变绿后，再跑一次全量冒烟：`pytest -q -k "smoke_"`

**禁止做：**
- 不要跳过/删除/放宽断言来"变绿"
- 不要在没变绿前写新功能/新文件/新依赖
- 不要凭记忆续写，以当前代码与测试结果为准

---

## 步骤 3：继续推进（全绿后自动衔接 codegen 流程）

全量冒烟通过后，自动衔接 codegen 工作流：

1. **找到 proposal**：按 `research/codedeepresearch/<upstream_name>/<upstream_name>_proposal.md` 的约定路径读取
   - `<upstream_name>` 从已有测试文件名推导（如 `test_smoke_openclaw_*.py` → `openclaw`）
   - 如果推导不出，问用户
2. **对照功能点清单**：把 proposal 中的功能点 vs 已通过的冒烟测试做交叉比对，确定下一个待实现的功能点
3. **按 codegen 流程继续**：接口设计 → 实现 → 冒烟测试 → 变绿 → 下一个

---

## 汇报格式

恢复阶段（步骤 1-2 完成后）：

```
## 断点恢复
- **中断前状态**：[上次做到哪个功能点，什么状态]
- **修复动作**：[改了什么文件，修了什么问题]（没有则写"无需修复"）
- **当前基线**：X 个冒烟测试全部通过 / Y 个仍失败
```

继续推进阶段（每完成一个功能点）：沿用 codegen 的汇报格式：
- 实现了什么
- 改了哪些文件
- 上游对齐情况
- 如何运行测试
